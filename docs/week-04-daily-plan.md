# 第四周按天执行计划：结构化输出与阶段项目一收口

## 总体安排

- 日期：2026-07-09（周四）至 2026-07-15（周三）
- 总投入：约 10.5 小时
- 工作日：每天 1–1.5 小时；周末：每天 2 小时
- 技术：Java 21、Spring Boot 4.1.0、Spring AI 2.0.0（`BeanOutputConverter`、`ChatClient.entity()`）、Jackson、Bean Validation（jakarta.validation）、JUnit 5 参数化测试
- 原则：本周是 16 周路线的**第一个硬性里程碑周**——第 4 周末要交付阶段项目一（结构化聊天服务）的完整形态。重心从"模板基础设施"转向"输出可信"：把 `/api/extract` 从 JSON 文本透传升级为经过解析与校验的 Java 对象，并用不少于 30 个案例的验收报告收口前四周成果。若前序有未达标项，按里程碑规则不允许靠堆新功能掩盖。

最终应得到：一个基于 `BeanOutputConverter` 的结构化抽取端点（返回 Java Record 而非 JSON 文本）、覆盖格式/语义/业务三类错误的校验与错误响应、解析失败的有限重试与失败样本记录、约 20–22 条离线参数化测试 + 8–10 条端到端用例构成的 30+ 案例验收报告，以及阶段项目一的收口结论。

> **与第三周的差异：** 第三周重心在开发（约 7 小时），测试只做轻量验证。本周开发与测试并重——编码约 5 小时，测试与验收报告约 2.5–3 小时。测试成本刻意做了拆分控制：**解析层与校验层的用例全部离线（伪造模型输出，不调 API）**，只有端到端验证和重试实验消耗真实调用。

> **承接第三周：** 直接偿还 Day21 技术债务清单第 1 条（抽取结果仅 JSON 文本透传）与第 3 条（A/B 用例区分度不足），并把 Day19 的注入结论（"可靠防线是输出侧收敛与校验"）从分类端点扩展到抽取端点。Day18 刻意采用的英文键名（`name`/`date`/`amount`）本周直接映射为 Record 字段。复用 `spring-ai-chat` 模块、`PromptTemplateService`、`GlobalExceptionHandler` 与 `ErrorResponse` 结构。

## 第 22 天：7 月 9 日，周四（1 小时）

**目标：** 建立结构化输出的概念框架，完成目标 Record 与错误分类设计。

**学习内容（约 30 分钟）：**

- 自然语言结果与机器可消费结果的差异：为什么"看起来像 JSON 的文本"不能直接给下游。
- Record / JSON Schema / Bean Validation 三者分工：Record 定义目标形状，`BeanOutputConverter` 从 Record 生成 Schema 并注入提示词，Bean Validation 在解析成功后做字段校验。
- 三类错误的区分（本周核心概念框架）：
  - **格式错误**：输出不是合法 JSON、缺字段、类型不对——解析层拦截；
  - **语义错误**：JSON 合法但字段内容不合理（如金额为负、日期是"明天"）——校验层拦截；
  - **业务校验错误**：字段各自合法但组合违反业务约束——业务规则层拦截。
- 失败处理链：修复提示（把解析错误回喂模型）→ 有限重试 → 降级为明确错误响应。

**执行任务（偏设计，约 30 分钟）：**

1. 在 `notes/week04.md` 中用自己的话写出三类错误的定义与各自的拦截层，并各举一个抽取场景的例子。
2. 设计抽取结果 Record（`name`/`date`/`amount`），**明确"未提及"的映射决定**：是映射为 `null`、`Optional` 还是保留哨兵值，写下理由（这个决定影响后续所有校验注解的写法）。
3. 设计错误响应的错误码方案：三类错误对应可区分的错误码，复用 `ErrorResponse` 结构。
4. 决定重试上限（建议 2 次，即最多 3 次调用）与失败样本的记录形式（建议：结构化日志或落地文件，含原始输出与失败原因）。

**产出：** `notes/week04.md` 初稿、Record 设计与"未提及"映射决定、三类错误的错误码方案、重试上限决定。

**验收：**

- [ ] 能各举一例说明格式错误、语义错误、业务校验错误的差异
- [ ] "未提及"如何映射到 Record 字段有明确决定与理由
- [ ] 重试上限是一个写下来的数字，而不是"再说"

## 第 23 天：7 月 10 日，周五（1.5 小时）

**目标：** 用 `BeanOutputConverter` 把 `/api/extract` 升级为返回 Java Record。

**执行任务：**

1. 定义 `ExtractResult` Record（按 Day22 设计），用 Spring AI 的 `BeanOutputConverter`（或 `ChatClient.entity()`）替代手工在模板里约定 JSON 格式。
2. 调整 `extract.yaml` 模板：格式约束部分改由转换器生成的 Schema 指令承担，模板保留字段语义说明与少样本；确认少样本输出与 Schema 一致。
3. `/api/extract` 响应体从 JSON 文本改为结构化对象（保留模型名、Token、耗时等元信息结构）。
4. 手动调用验证一次正常抽取，确认响应是解析后的对象而非透传文本。
5. **红线自查：** 全程不允许出现字符串截取或正则"猜"JSON 的代码（如 `substring` 找大括号、正则抠代码块）。若模型输出带 markdown 代码块导致解析失败，交给 Day25 的重试机制处理，而不是在解析前偷偷清洗。

**产出：** `ExtractResult` Record、基于 `BeanOutputConverter` 的抽取链路、调整后的抽取模板、一次成功调用记录。

**验收：**

- [ ] `/api/extract` 返回结构化对象，不再透传 JSON 文本
- [ ] 格式约束由转换器 Schema 承担，模板不再手工描述 JSON 结构
- [ ] 代码中无字符串截取/正则猜 JSON
- [ ] 解析失败此时允许直接报错（重试是 Day25 的事，今天不做）

## 第 24 天：7 月 11 日，周六（2 小时）

**目标：** 落地三层校验与三类错误的可区分错误响应。

**执行任务：**

1. 在 pom 中补充 `spring-boot-starter-validation` 依赖（当前 pom 只有 `webmvc`，不含 Bean Validation 实现），然后为 `ExtractResult` 加校验注解：必填/可空（按"未提及"映射决定）、长度上限、格式约束（如金额的数值性、日期的格式）。
2. 增加至少一条**字段间业务约束**校验（例如：金额存在时日期不得缺失，或按你的业务语义自定），实现放在服务层校验器而非 Controller。
3. 在 `GlobalExceptionHandler` 中把三类错误映射为可区分的错误码与 HTTP 状态：
   - 格式错误（解析异常）；
   - 语义错误（Bean Validation 违例）；
   - 业务校验错误（自定义校验器违例）。
4. **明确对象流出规则：** 只有通过全部校验的对象才能作为成功响应返回；任何一层失败都走 `ErrorResponse`，不允许返回"部分可用"的对象。
5. 手动构造两三个非法输入（如让模型抽取一段必然缺字段的文本）验证错误响应形态。

**产出：** 带校验注解的 Record、业务约束校验器、三类错误的异常映射、手动验证记录。

**验收：**

- [ ] 三类错误返回可区分的错误码，调用方能判断错在哪一层
- [ ] 下游只接收通过校验的对象，无"部分可用"泄漏
- [ ] 业务约束校验在服务层，Controller 不含校验逻辑
- [ ] 校验失败返回结构化错误，而非堆栈

## 第 25 天：7 月 12 日，周日（2 小时）

**目标：** 实现解析失败的有限重试、修复提示与降级响应，并使其可观测。

**执行任务：**

1. 实现有限重试：解析（或校验）失败时，构造**修复提示**——把上一次的原始输出与具体失败原因回喂给模型，要求其修正——重试次数不超过 Day22 定下的上限。
2. 记录失败样本：每次失败保存原始输出、失败原因、所属错误类别、重试轮次（这些样本是 Day27 验收报告与后续评估的素材）。
3. 重试耗尽后的降级：返回明确的"无法可靠解析"错误响应（独立错误码），**不返回**最后一次的原始文本。
4. 可观测性：响应元信息或日志中记录本次请求实际发生的重试次数与最终结果，确保"重试有限且可观测"可以被验证而不是口头承诺。
5. 手动诱导一次解析失败（例如临时在模板里加干扰指令，或用极端输入），观察完整的"失败 → 修复提示 → 重试 → 成功/降级"链路并记录。

**产出：** 有限重试与修复提示实现、失败样本记录机制、降级错误响应、一次完整失败链路的观察记录。

**验收：**

- [ ] 重试有硬上限，不存在无限重试路径（防止无限烧钱）
- [ ] 每次失败的原始输出与原因被记录，事后可查
- [ ] 重试次数可从日志或响应元信息中读出
- [ ] 重试耗尽返回明确错误，不透传原始文本

## 第 26 天：7 月 13 日，周一（1.5 小时）

**目标：** 离线参数化测试——约 20–22 条用例，全部不调模型（本周测试成本的大头，API 成本为零）。

> **拆分依据：** 解析层与校验层的行为验证不需要真实模型——拿伪造的"模型输出"字符串喂给转换与校验链路即可，性质与 Day20 的 13 条渲染单测相同，参数化后加一条用例的边际成本只有几分钟。真实调模型的用例集中在 Day27，控制在 8–10 条。

**执行任务：**

1. 为解析与校验链路搭参数化测试骨架（JUnit 5 `@ParameterizedTest`），输入为伪造的模型输出字符串，断言解析/校验结果或异常类别。
2. 按五类场景铺用例，目标 20–22 条：
   - **缺字段**（约 4 条）：缺单个字段、缺多个字段、空 JSON、字段名拼错；
   - **错类型**（约 4 条）：金额为字符串、日期为数字、字段值为数组/对象等；
   - **未知枚举/非法值**（约 4 条）：含"未提及"哨兵的各种位置组合、超出约定的取值；
   - **超长文本**（约 3 条）：单字段超长、整体超长、超长但合法（应通过）；
   - **事实冲突/业务违例**（约 3 条）：违反 Day24 业务约束的字段组合；
   - **正常样本**（约 3–4 条）：全字段齐全、部分"未提及"、边界长度——确认校验不误杀。
3. 每条用例断言到**错误类别**（格式/语义/业务），不只断言"抛了异常"。
4. 确认默认 `mvn test` 全绿且零 API 调用（沿用 Day20 修复后的 `excludedGroups` 机制）。

**产出：** 20–22 条离线参数化测试、按五类场景组织的用例清单（直接作为验收报告素材）。

**验收：**

- [ ] 用例数不少于 20，覆盖五类场景与正常样本
- [ ] 每条用例断言到具体错误类别，而非笼统的异常
- [ ] 默认 `mvn test` 不调用付费 API
- [ ] 正常样本全部通过，无校验误杀
