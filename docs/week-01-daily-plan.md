# 第一周按天执行计划：LLM 工程基础

## 总体安排

- 日期：2026-06-18（周四）至 2026-06-24（周三）
- 总投入：约 9.5 小时
- 工作日：每天 1–1.5 小时；周末：每天 2 小时
- 技术：Java 21、Maven、Java HTTP Client、Jackson、JUnit 5、WireMock
- 原则：本周理解模型 API 底层行为，暂不使用 Spring AI 封装

最终应得到一个可运行的 `llm-basics` 项目、OpenAI API 客户端、Token/耗时/错误记录、至少 20 条测试输入、15 次参数实验、6 类故障测试及一份周报。

## 第 1 天：6 月 18 日，周四（1 小时）

**目标：** 建立 LLM API 的概念框架。

**学习内容：**

- Token 与字符、单词的区别
- 上下文窗口包含的消息和模型输出
- System、User、Assistant 消息的职责
- Temperature、最大输出长度等参数
- Chat Model 与 Embedding Model 的差异

**执行任务：**

1. 阅读 OpenAI 文档中的文本生成、Token 和模型参数章节。
2. 用自己的语言解释 Token、上下文窗口、Temperature 和 Embedding。
3. 准备 5 个实验问题：2 个确定性问题、2 个开放式问题、1 个容易诱发幻觉的问题。

**产出：**

- `notes/day-01-llm-basics.md`
- 5 条实验问题
- Chat Model 与 Embedding Model 对比表

**验收：**

- [ ] 不把低 Temperature、事实准确和无幻觉混为一谈
- [ ] 不查资料也能解释上下文窗口

## 第 2 天：6 月 19 日，周五（1 小时）

**目标：** 搭建 Java 原生模型调用项目。

**执行任务：**

1. 创建 `llm-basics` Maven 项目。
2. 配置 Java 21、Jackson、JUnit 5 和 WireMock。
3. 从 `OPENAI_API_KEY` 环境变量读取密钥。
4. 定义最小请求和响应对象。
5. 使用 Java HTTP Client 发出一次请求。
6. 输出模型回答、HTTP 状态和总耗时。

**建议结构：**

```text
llm-basics/
├─ pom.xml
├─ src/main/java/
│  ├─ client/OpenAiClient.java
│  ├─ model/ChatRequest.java
│  ├─ model/ChatResponse.java
│  └─ App.java
└─ src/test/java/
```

**产出：** 可编译项目、第一次 API 调用记录、`.gitignore` 和环境变量说明。

**验收：**

- [ ] 密钥未进入源码、配置或 Git
- [ ] 入口类不负责拼装 HTTP 请求
- [ ] 缺少密钥时返回明确错误

## 第 3 天：6 月 20 日，周六（2 小时）

**目标：** 完成可复用、可观测的 API 客户端。

**执行任务：**

1. 增加连接超时、请求超时、状态判断和 JSON 解析异常处理。
2. 记录模型、输入/输出/总 Token、耗时、HTTP 状态和成功状态。
3. 定义认证失败、限流、服务端错误、网络/超时和响应格式错误。
4. 编写正常响应解析测试。

**产出：** `OpenAiClient`、请求/响应 DTO、统一异常分类、单元测试和日志样例。

**验收：**

- [ ] 上层代码不依赖原始 JSON
- [ ] 日志不输出 API Key
- [ ] 能读取 Token 和耗时
- [ ] 能从异常判断故障类别

## 第 4 天：6 月 21 日，周日（2 小时）

**目标：** 完成参数对照实验。

| 组别 | 任务 | Temperature | 输出长度 | 次数 |
|---|---|---:|---:|---:|
| A | 确定性问答/分类 | 0 | 较短 | ≥5 |
| B | 相同确定性任务 | 0.7 | 较短 | ≥5 |
| C | 开放式生成 | 1.0 | 较长 | ≥5 |

**执行任务：**

1. 固定模型和主要 Prompt，只改变目标参数。
2. 每组至少执行 5 次。
3. 记录输出、任务成功状态、Token、耗时和事实错误。
4. 加入容易诱发幻觉的问题。
5. 比较 Temperature 为 0 时的稳定性和正确性。

**产出：** `experiments/parameter-results.csv`、至少 15 次结果、参数对比表和初步结论。

**验收：**

- [ ] 对照实验未同时随意修改多个变量
- [ ] “结果更好”有明确判定标准
- [ ] 能用数据说明低随机性不等于事实正确

## 第 5 天：6 月 22 日，周一（1 小时）

**目标：** 扩大样本并分析 Token、延迟和成本。

**执行任务：**

1. 将输入扩充至至少 20 条，覆盖事实问答、摘要、分类、抽取、开放生成、歧义和无法确认的问题。
2. 汇总各类任务平均 Token 和耗时。
3. 根据当天 OpenAI 官方价格计算实验成本，并记录模型、日期和价格来源。
4. 分析输入变长对 Token 和延迟的影响。

**产出：** 20 条测试输入、`experiments/task-results.csv` 和成本汇总表。

**验收：**

- [ ] 成本计算可追溯
- [ ] 区分输入与输出 Token
- [ ] 未用单次结果冒充稳定性能结论

## 第 6 天：6 月 23 日，周二（1.5 小时）

**目标：** 完成错误与故障模拟。

**执行任务：**

1. 使用 WireMock 模拟 401、429、500/503、超时、非法 JSON 和缺少关键字段。
2. 验证每类故障映射到正确异常。
3. 只为适合重试的故障设置有限重试。
4. 验证 401 和格式错误不会被盲目重试。

**产出：** WireMock 测试、至少 6 类失败场景和错误处理决策表。

**验收：**

- [ ] 测试不调用真实付费 API
- [ ] 401、429、5xx 和超时可区分
- [ ] 重试次数有限
- [ ] 非法响应不会成为正常回答

## 第 7 天：6 月 24 日，周三（1 小时）

**目标：** 完成验收、总结并衔接第二周。

**执行任务：**

1. 运行全部自动化测试。
2. 检查 Git 中是否包含密钥或敏感响应。
3. 整理实验数据。
4. 总结 Token、上下文窗口、Embedding、Temperature、重试和成本因素。
5. 列出原生客户端的重复代码，作为第二周引入 Spring AI 的对照。

**产出：**

- `reports/week-01-report.md`
- 测试结果与参数实验结论
- 技术债务清单
- 第二周准备清单

**验收：**

- [ ] 一条 Maven 命令可以构建并测试
- [ ] 完成至少 20 条输入和 15 次参数实验
- [ ] 全部故障模拟通过
- [ ] 能区分调用失败与模型回答错误
- [ ] 能用实验回答“Temperature 为 0 是否仍会幻觉”
