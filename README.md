# Spring AI 企业应用开发：16 周学习计划

> 面向已经掌握 Spring Boot、REST API、Maven、数据库和基础 Docker，并使用过大模型 API 的 Java 开发者。

## 1. 学习目标

本计划周期为 **16 周**，每周投入 **8–10 小时**，总投入约 **128–160 小时**。目标不是停留在“能调用模型”，而是能够独立设计、实现、评测和部署一个具备基本生产属性的 Spring AI 应用。

完成后应具备以下能力：

- 使用 Spring AI 完成同步、流式、结构化输出、多轮会话和工具调用。
- 理解 Embedding、向量检索与 RAG 的完整数据流，而不是机械拼接 API。
- 建立可重复的 RAG 评测集，用数据比较检索策略、模型质量、延迟和成本。
- 处理提示注入、越权检索、敏感信息泄漏、限流、超时和模型故障。
- 部署一个带文档导入、权限过滤、引用溯源、监控和自动化测试的企业知识库助手。

## 2. 学习边界

### 本阶段重点

- Spring AI 核心抽象与工程实践
- OpenAI 模型调用
- Prompt、Structured Output、Chat Memory、Tool Calling
- PostgreSQL + pgvector
- RAG 数据工程、检索优化与评测
- 安全、权限、可观测性、可靠性和部署

### 暂不作为主线

- 模型训练与微调
- 复杂多智能体框架
- GPU 集群和大模型推理平台建设
- 只为演示效果设计、无法评测的“玩具聊天机器人”

这些内容可以在完成本计划后继续学习。过早进入多智能体编排，通常只是把一个不可控模型扩展为多个不可控模型，并不会自动提高系统质量。

## 3. 技术环境

建议使用：

| 类别 | 选择 | 说明 |
|---|---|---|
| JDK | Java 21 | 使用当前 LTS 版本 |
| 构建工具 | Maven | 建议使用 Maven Wrapper |
| Web 框架 | Spring Boot | 通过 Spring Initializr 选择兼容版本 |
| AI 框架 | Spring AI | 以官方参考文档中的当前稳定版为准 |
| 模型服务 | OpenAI API | 主模型和 Embedding 模型分开配置 |
| 数据库 | PostgreSQL | 保存业务数据、会话和文档元数据 |
| 向量扩展 | pgvector | 保存与检索 Embedding |

| 可观测性 | Actuator + Micrometer | 记录业务和模型调用指标 |

Spring AI 更新速度较快，学习时不要从随机博客复制版本号或旧 API。应从以下两个入口确认兼容关系：

- [Spring Initializr](https://start.spring.io/)
- [Spring AI Reference](https://docs.spring.io/spring-ai/reference/)

## 4. 密钥、成本与安全基线

### API Key

禁止把密钥写入 Java 源码、`application.yml`、测试快照或 Git 提交。开发环境使用环境变量：

```powershell
$env:OPENAI_API_KEY="your-api-key"
```

配置文件只引用变量：

```yaml
spring:
  ai:
    openai:
      api-key: ${OPENAI_API_KEY}
```

生产环境应使用云平台 Secret Manager、Kubernetes Secret 或等价密钥管理系统，并设置密钥轮换策略。

### 成本控制

- 为开发和生产分别设置 API 项目、预算与告警。
- 默认使用满足任务质量的较小模型；只有评测证明必要时才升级模型。
- 记录每次请求的输入 Token、输出 Token、模型、耗时和业务场景。
- 对 Prompt、对话历史、检索上下文设置明确上限。
- 本地测试优先使用固定响应或 Mock，禁止单元测试无控制地调用付费 API。
- 每周根据供应商价格页重新计算预算，不在代码或文档中长期硬编码价格。

### 安全基线

- 模型输出默认视为不可信数据，必须校验后才能执行。
- 工具调用实行最小权限，不向模型暴露数据库写权限、Shell 或任意 HTTP 请求能力。
- 权限过滤必须发生在检索层，不能只依赖 Prompt 告诉模型“不要泄露”。
- 日志不得记录 API Key、完整敏感文档或未经脱敏的用户输入。

## 5. 每周固定节奏

每周建议分配：

- 理论与官方文档：约 2 小时
- 编码与项目实践：4–5 小时
- 测试、评测与复盘：2–3 小时

建议将学习拆成 3–4 次完成，避免把 8–10 小时集中在一天。每周编码时间优先用于完成主线任务；如果主线任务提前完成，剩余时间用于补充失败场景、扩大评测集和重构，而不是继续堆叠新框架。

每周都应提交代码和一份简短复盘，至少回答：

1. 本周实现的能力是什么？
2. 哪些结论有测试或数据支持？
3. 最大失败模式是什么？
4. 下周应保留、修改或删除什么？

## 6. 16 周详细路线

### 第 1 周：LLM 工程基础

**时间预算：** 理论 2 小时，编码 4 小时，实验与复盘 2–4 小时。

**按天执行：** [第一周按天执行计划](docs/week-01-daily-plan.md)

**理论主题**

- Token、上下文窗口、系统消息和生成参数
- Temperature 与结果随机性
- 幻觉的成因、可缓解范围与不可消除性
- Embedding 与生成模型的职责差异
- 延迟、Token 和成本之间的约束关系

**编码任务**

- 使用 Java HTTP Client 或 OpenAI 官方 Java SDK 调用一次 Chat API。
- 分别测试确定性任务和开放式任务的不同参数。
- 记录请求模型、输入/输出 Token、总耗时和异常信息。
- 为 API 调用封装超时及错误处理，避免 Controller 直接拼接 HTTP 请求。
- 对至少 3 组 Temperature、输出长度和 Prompt 组合进行重复实验，每组不少于 5 次。
- 使用 WireMock 或等价方案模拟 401、429、5xx、超时和非法响应。

**交付物**

- `llm-basics` 示例或独立模块。
- 至少 20 条测试输入及实验记录，并附参数对照表。
- 一页关于 Token、延迟和成本的总结。

**验收标准**

- 能解释 Token、上下文窗口和 Embedding，且不把 Embedding 当作答案生成器。
- 密钥未进入 Git。
- 能区分网络错误、限流、认证失败和模型正常但回答错误。

**推荐资料**

- [OpenAI API 文档](https://platform.openai.com/docs/)
- [OpenAI API 参考](https://platform.openai.com/docs/api-reference)

**思考题**

> 如果把 Temperature 调成 0，是否就不会产生幻觉？设计实验验证，而不是凭直觉回答。

### 第 2 周：Spring AI 入门

**时间预算：** 理论 2 小时，编码 5 小时，测试与复盘 1–3 小时。

**按天执行：** [第二周按天执行计划](docs/week-02-daily-plan.md)

**理论主题**

- Spring AI 的 `ChatModel`、`ChatClient` 和自动配置
- Prompt、Message、Response 的职责
- 同步调用与流式调用的差异
- 框架抽象带来的供应商解耦及其边界

**编码任务**

- 使用 Spring Initializr 创建 Spring Boot 项目并接入 Spring AI OpenAI Starter。
- 实现同步聊天、流式聊天和动态参数调用。
- 使用配置属性区分模型名称、超时和生成参数。
- 为外部调用增加集成测试；普通单元测试使用 Stub 或 Mock。
- 为流式接口处理客户端取消、上游异常和连接超时。
- 用序列图说明 Controller、`ChatClient`、`ChatModel` 与供应商 API 的调用关系。

**交付物**

- `/api/chat` 和 `/api/chat/stream` 两个接口。
- 配置说明和最小运行示例。
- 同步、流式、异常及客户端取消场景测试。

**验收标准**

- 流式接口能持续输出，而不是等待完整结果后伪装成流式。
- Controller 不包含模型供应商 SDK 细节。
- 能说明 Spring AI 抽象没有消除不同模型能力差异。

**推荐资料**

- [Spring AI Chat Model API](https://docs.spring.io/spring-ai/reference/api/chatmodel.html)
- [Spring AI ChatClient](https://docs.spring.io/spring-ai/reference/api/chatclient.html)

**思考题**

> 哪些参数应由业务固定，哪些可以由用户控制？如果用户可直接设置最大输出长度，会产生什么风险？

### 第 3 周：提示词工程与可测试模板

**时间预算：** 理论 2 小时，编码 4–5 小时，评测与复盘 2–3 小时。

**按天执行：** [第三周按天执行计划](docs/week-03-daily-plan.md)

**理论主题**

- System、User 和 Assistant 消息的作用
- 零样本、少样本、任务分解与上下文约束
- Prompt Template 与变量边界
- 提示注入和数据指令混淆

**编码任务**

- 为摘要、分类和信息抽取建立独立模板。
- 将模板版本、输入变量和预期输出纳入测试数据。
- 对缺失变量、超长输入和恶意指令建立失败测试。
- 比较“堆砌自然语言规则”和结构化上下文的差异。
- 为模板增加版本号、用途、适用模型和变更原因元数据。
- 设计 A/B 对照：固定模型与参数，只改变 Prompt，统计任务成功率。

**交付物**

- 三类可复用 Prompt 模板。
- 至少 30 个固定测试案例，包含正常、边界、歧义和攻击输入。
- Prompt 变更记录。

**验收标准**

- Prompt 不散落在 Controller 和 Service 中。
- 模板输入经过长度、空值和类型检查。
- Prompt 修改后能通过固定测试集发现回归。

**推荐资料**

- [Spring AI Prompts](https://docs.spring.io/spring-ai/reference/api/prompt.html)
- [OWASP LLM Prompt Injection](https://genai.owasp.org/llmrisk/llm01-prompt-injection/)

**思考题**

> 为什么“忽略用户恶意指令”这句 Prompt 不能构成可靠的安全边界？

### 第 4 周：结构化输出

**时间预算：** 理论 1.5–2 小时，编码 5 小时，测试与复盘 1.5–3 小时。

**理论主题**

- 自然语言结果与机器可消费结果的差异
- Java Record/DTO、JSON Schema 和 Bean Validation
- 格式错误、语义错误与业务校验错误
- 重试、修复提示和降级策略

**编码任务**

- 将模型输出映射为 Java Record。
- 对字段类型、枚举、长度和必填项进行校验。
- 对解析失败设置有限重试，并记录失败样本。
- 实现无法可靠解析时的明确错误响应。
- 增加字段间业务约束校验，并区分格式错误、语义错误与业务错误。
- 使用参数化测试覆盖缺字段、错类型、未知枚举、超长文本和事实冲突。

**交付物**

- 阶段项目一：结构化聊天服务。
- 分类或工单提取 API。
- 正常、边界和错误样本测试。
- 一份不少于 30 个案例的阶段验收报告。

**验收标准**

- 不使用字符串截取或正则表达式“猜测”JSON。
- 重试次数有限且可观测。
- 下游业务只接收通过校验的对象。

**推荐资料**

- [Spring AI Structured Output](https://docs.spring.io/spring-ai/reference/api/structured-output-converter.html)

**思考题**

> 模型返回格式合法但业务事实错误时，结构化输出解决了什么，又没有解决什么？

### 第 5 周：会话记忆

**时间预算：** 理论 2 小时，编码 4–5 小时，测试与复盘 2–3 小时。

**理论主题**

- Chat Memory 与业务会话持久化的区别
- Message Window、Token 上限和历史裁剪
- 会话隔离、并发更新和数据生命周期
- 摘要记忆可能导致的信息损失

**编码任务**

- 使用会话 ID 实现多轮对话。
- 为不同用户隔离历史记录。
- 设置最大消息数或 Token 预算。
- 对会话过期、并发请求和服务重启进行测试。
- 将会话状态持久化到数据库，并设计乐观锁或等价并发控制。
- 比较消息窗口裁剪与历史摘要两种策略对质量、延迟和 Token 的影响。

**交付物**

- 多轮会话 API。
- 会话数据模型和生命周期说明。
- 会话隔离、并发、过期、恢复和裁剪策略对比报告。

**验收标准**

- 用户 A 无法读取用户 B 的历史。
- 历史不会无限增长。
- 能解释“模型记住了”只是应用重新提交上下文，并非模型参数发生改变。

**推荐资料**

- [Spring AI Chat Memory](https://docs.spring.io/spring-ai/reference/api/chat-memory.html)

**思考题**

> 裁剪最旧消息和摘要压缩分别会破坏哪些信息？如何构造回归测试？

### 第 6 周：工具调用

**时间预算：** 理论 2 小时，编码 5 小时，安全测试与复盘 1–3 小时。

**理论主题**

- Tool Calling 的调用流程和参数 Schema
- 模型决策与应用授权的区别
- 幂等、超时、重试、审计和人工确认
- 工具结果污染与间接提示注入

**编码任务**

- 实现天气查询、订单查询或知识库统计等只读工具。
- 对参数进行校验，并设置调用超时。
- 为有副作用的模拟工具加入确认令牌和幂等键。
- 记录工具名称、参数摘要、调用者、耗时和结果状态。
- 增加工具白名单、调用次数上限和循环调用保护。
- 测试伪造身份、参数注入、重复提交、工具超时和恶意工具返回值。

**交付物**

- 阶段项目二：工具调用助手。
- 工具权限矩阵和失败处理说明。
- 不少于 20 个工具调用测试场景及审计记录样例。

**验收标准**

- 模型不能调用未注册工具。
- 工具层独立执行鉴权，不能相信模型声称的用户身份。
- 重试不会导致重复写入。

**推荐资料**

- [Spring AI Tool Calling](https://docs.spring.io/spring-ai/reference/api/tools.html)

**思考题**

> “模型选择调用工具”是否等价于“系统授权执行工具”？如果混为一谈，最严重的后果是什么？

### 第 7 周：Embedding 与向量检索

**时间预算：** 理论 2 小时，编码 4–5 小时，实验与复盘 2–3 小时。

**理论主题**

- Embedding、向量维度、相似度与距离
- Top-K、阈值和元数据过滤
- 召回率与精确率的基本权衡
- 向量模型变更为何需要重新索引

**编码任务**

- 使用 Docker 启动 PostgreSQL + pgvector。
- 导入一批带部门、权限和来源元数据的文档片段。
- 对固定查询执行向量检索并观察不同 Top-K 和阈值。
- 保存 Embedding 模型、维度和索引版本。
- 使用不少于 100 个文档片段构造最小检索语料，并人工标注相关性。
- 比较余弦距离、不同 Top-K、阈值及元数据过滤对结果的影响。

**交付物**

- 可重复执行的数据库初始化方案。
- 至少 30 个查询及相关文档标注。
- 参数实验表。

**验收标准**

- 检索结果包含文本、相似度和元数据。
- 能解释高相似度不等于事实相关。
- 更换 Embedding 模型时不会静默混用旧向量。

**推荐资料**

- [Spring AI Vector Databases](https://docs.spring.io/spring-ai/reference/api/vectordbs.html)
- [pgvector](https://github.com/pgvector/pgvector)

**思考题**

> Top-K 越大是否必然提高最终答案质量？分别从召回、噪声、上下文长度和成本分析。

### 第 8 周：基础 RAG

**时间预算：** 理论 2 小时，编码 5 小时，测试与复盘 1–3 小时。

**理论主题**

- 文档读取、清洗、切分、Embedding、检索和生成链路
- Chunk 大小、重叠和语义边界
- Grounding、引用与无答案处理
- 离线索引和在线查询的职责分离

**编码任务**

- 导入 Markdown、TXT 或 PDF 文档。
- 完成文档切分、向量化和存储。
- 查询时检索相关片段并构造受控上下文。
- 返回答案、引用文档、片段位置和“证据不足”状态。
- 为索引和查询链路增加结构化日志及耗时分段统计。
- 增加文档为空、格式异常、重复上传、无相关结果和超长上下文测试。

**交付物**

- 基础知识库问答 API。
- 索引流程图和查询流程图。
- 至少 40 个问题的初始测试集，其中无答案问题不少于 10 个。

**验收标准**

|---|---|---|---|

## 10. 常见误区

1. **把能调用 API 当作掌握 Spring AI。** 这只证明网络和密钥可用，没有证明系统可维护。
2. **只调 Prompt，不建立评测集。** 没有固定样本和指标，就无法判断修改是优化还是随机波动。
3. **把 RAG 等同于向量数据库。** 数据清洗、切分、权限、引用、评测和更新机制通常更困难。
4. **认为 Temperature 为 0 就不会幻觉。** 低随机性不等于事实正确，只是输出更稳定。
5. **把模型当作权限系统。** 模型不能替代鉴权，未授权数据不应进入上下文。
6. **在单元测试中直接调用付费模型。** 这会导致测试慢、不稳定且成本不可控。
7. **无限增加对话历史和检索片段。** 更多上下文可能增加噪声、延迟和成本。
8. **为了“解耦”抹平所有供应商特性。** 合理抽象业务接口，但必须显式处理能力差异。
9. **过早进入复杂 Agent。** 如果单次检索、工具权限和失败处理尚不可靠，多智能体只会增加调试难度。
10. **只做成功路径演示。** 企业应用的主要工作恰恰是处理错误、权限、质量漂移和成本。
